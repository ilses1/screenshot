## 重构目标
- 保持现有功能和行为完全不变（快捷键、截图、历史、贴图等都不改）。
- 优化主进程（“后端”）和渲染进程（“前端”）的代码结构，让职责更清晰、文件更易维护。
- 为后续加功能（如更多设置项、更复杂编辑器）打好架构基础。

## 一、主进程（后端）结构优化
**目标：拆分大而全的 index.ts，让每个模块只做一件事。**

1. 提取公共类型与常量
- 新建一个公共类型文件，例如 `src/common/types.ts`：
  - 定义 `AppConfig`、`ScreenshotRecord` 等通用类型。
- 新建 IPC 通道常量文件，例如 `src/common/ipcChannels.ts`：
  - 统一定义 `'settings:get'`、`'settings:update'`、`'capture:save-image'`、`'history:list'` 等通道字符串，避免在各处硬编码。

2. 拆分配置与历史管理
- 把 `loadConfig` / `saveConfig` 抽到 `src/main/configService.ts`：
  - 提供 `loadConfig(): AppConfig`、`saveConfig(config: AppConfig)`，并内聚 `getConfigPath` 等细节。
- 把 `loadHistory` / `saveHistory` 及内存中的 `history` 状态抽到 `src/main/historyService.ts`：
  - 提供 `initHistory()`、`getHistory()`、`addRecord()`、`clearHistory()` 等接口。
- 主入口只通过 service 接口使用这些能力，不直接碰文件读写逻辑。

3. 拆分窗口管理
- 新建 `src/main/windowManager.ts`，集中管理：
  - `createMainWindow`
  - `createCaptureWindow`
  - `createEditorWindow`
  - `createPinWindow`（当前 pin:last 里构造贴图窗口的逻辑）
- 统一管理 `BrowserWindow` 的创建参数和事件（如 `closed`、`did-finish-load`），主入口只调用这些函数。

4. 拆分托盘和快捷键注册
- 新建 `src/main/trayAndShortcuts.ts`：
  - `createTray(config: AppConfig)`：内部处理图标、菜单、点击回调（调用 windowManager 的方法）。
  - `registerShortcuts(config: AppConfig)`：内部调用 `globalShortcut.register`，输出日志，用于响应设置变更。
- 这样设置更新时只要重新调用 `registerShortcuts`，主入口逻辑更简洁。

5. 拆分 IPC 注册
- 新建 `src/main/ipcHandlers.ts`：
  - 提供 `registerIpcHandlers(deps)`，其中 `deps` 注入 configService、historyService、windowManager 等需要用到的能力。
  - 内部使用公共的 `ipcChannels` 常量，集中维护所有 IPC 通道及其处理逻辑。

6. 精简主进程入口 index.ts
- 入口文件只做：
  - 初始化配置与历史：`configService.loadConfig()`、`historyService.initHistory()`；
  - 创建主窗口和托盘：`windowManager.createMainWindow()`、`createTray(config)`；
  - 注册全局快捷键和 IPC：`registerShortcuts(config)`、`registerIpcHandlers(...)`；
  - 处理 `app.whenReady`、`app.on('activate')`、`app.on('will-quit')` 等生命周期事件。

## 二、渲染进程（前端）结构优化
**目标：让设置页、截图页、编辑器页各自职责更清晰，减少内联 HTML 和重复逻辑。**

1. 设置页（index.html / main.ts）
- 新建 `src/renderer/api.ts`：
  - 封装对 `window.api` 的调用，提供：
    - `getSettings()`、`updateSettings()`
    - `getHistory()`、`clearHistory()`、`pinLast()`
  - 使用前面公共的 `AppConfig`、`ScreenshotRecord` 类型，保证前后端类型一致。
- 在 `main.ts` 中：
  - 将现在大块 `app.innerHTML = ...` 的模板拆成一个 `renderSettingsUI(root: HTMLElement)` 函数，方便以后改为更复杂的布局。
  - 初始化阶段逻辑拆成几个小函数：`bindEvents()`、`initSettings()`、`initHistory()`，提高可读性。

2. 截图页（capture.html / capture.ts）
- 在 `capture.ts` 内部引入更清晰的状态结构：
  - 定义 `SelectionState`，管理起点/终点坐标和是否正在选择。
  - 将 `draw` 与“画蒙层 + 高亮选区”的逻辑拆成独立函数，便于以后加功能（例如显示选区尺寸）。
- 在 `capture.html` 中：
  - 增加一个固定位置的提示文本元素，例如左上角：
    - “按住左键拖动选择区域，松开完成截图，Esc/右键取消”。
  - 保持当前截图完成和取消逻辑不变，只是增加 UI 提示（不改变功能）。

3. 编辑器页（editor.html / editor.ts）
- 按同样思路，对编辑器中的工具状态（画笔、矩形、撤销等）做适度整理：
  - 抽出一个 `EditorState` 对象，集中管理当前工具、颜色、线宽。
  - 将事件绑定（鼠标/键盘）与渲染逻辑分开，减少重复代码。
- 暂时不改变编辑功能本身，只做结构上的整理。

## 三、前后端接口与安全边界
**目标：前后端之间的交互清晰、可追踪，为将来安全加固打基础。**

1. 统一 IPC 通道
- 所有 IPC 调用使用 `ipcChannels` 文件中导出的常量：
  - 避免手写字符串带来的拼写错误。
  - 未来新增功能时，可以在一个文件中一眼看到所有前后端接口。

2. 预留安全优化空间（可选，不在本轮改动中强制执行）
- 当前部分窗口（截图、编辑器）开启了 `nodeIntegration`，后续可以考虑：
  - 使用 preload + `contextBridge` 暴露受限 API 替代直接在 renderer 中引入 `electron` 模块；
  - 这一点会涉及到较大改动，为避免影响现有功能，本轮只在结构上预留位置，不改变行为。

## 四、不影响功能的验证方式
- 启动应用：`npm run dev`。
- 依次验证：
  - 托盘菜单“截图”“设置”“退出”是否行为一致；
  - 快捷键（当前为 F2 或设置中显示的快捷键）是否正常进入截图；
  - 截图选择、取消（Esc/右键）行为是否和现在一致；
  - 设置保存/加载、历史列表、贴最近截图是否正常工作。
