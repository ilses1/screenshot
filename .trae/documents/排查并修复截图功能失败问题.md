## 现状与可能原因
- 当前截图流程代码：托盘菜单/全局快捷键 → `createCaptureWindow` → 加载 `capture.html` → 执行 `capture.ts` 中的 `captureScreen()` → 选择区域 → 复制到剪贴板（如开启自动保存则写文件）。
- 从代码来看，整体流程是合理的，但用户感知为“截图不成功”，常见几类可能：
  - 截图窗口其实已经弹出，但是全透明叠加层，只看到鼠标形状变化，误以为没有进入截图状态。
  - `capture.ts` 里 `desktopCapturer.getSources` 或 `screen.getPrimaryDisplay()` 等调用在运行时报错，被 `catch` 后直接 `window.close()`，窗口一闪而过或根本没看到。
  - 只复制到了剪贴板（默认 `autoSaveToFile = false`），用户在磁盘上找不到图片，误判为“没截图成功”。
  - 多屏/缩放场景中，截图区域与实际屏幕像素不匹配，导致截出的图是错位或空白，看起来像失败。

- 验证托盘菜单：
  - 点击托盘“截图”菜单项时，确认日志中有 `createCaptureWindow` 被调用的记录。
  - 如窗口创建但立刻被关闭，结合 DevTools 日志进一步定位是前端报错还是我们主动 `window.close()`。

## 步骤三：针对截图逻辑的可能修改方案
（以下为准备实施的改动方向，待你确认后再真正修改代码）
- 提升用户可感知度：
  - 在进入截图模式时，适当加一层半透明蒙层和提示文字（例如左上角显示“拖动选择截图区域，Esc 取消”），避免你不知道自己已经在截图模式。
- 优化 `capture.ts` 的稳定性：
  - 如果发现 `desktopCapturer.getSources` 在渲染进程不可用或报安全错误，则改为：渲染进程通过 IPC 调用主进程的 `CAPTURE_GET_SOURCES`，由主进程调用 `desktopCapturer` 并把 dataURL 形式的缩略图返回给前端。
  - 对于空返回、0 宽高、`thumbnail.isEmpty()` 等情况，给出友好错误提示，而不是直接关闭窗口。
- 处理多屏与缩放问题：
  - 使用 `screen.getPrimaryDisplay()` 的 `bounds` 和 `scaleFactor` 计算实际像素尺寸，确保 `thumbnailSize` 与画布尺寸一致，避免局部空白或偏移。
